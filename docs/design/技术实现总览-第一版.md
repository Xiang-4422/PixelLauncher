# PixelLauncher 技术实现总览（第一版）

本文档用于帮助后续工程师快速理解 PixelLauncher 当前的真实技术实现。  
它关注“代码如何组织、数据如何流动、页面如何渲染、哪些边界不能轻易打破”，不重复产品设计 PRD。

若与旧文档或自动生成内容冲突，以当前代码为准。

## 1. 应用定位与运行时架构

PixelLauncher 当前不是 Compose 应用，也不是多 Activity / 多 Fragment 的常规页面架构。  
整个应用的主干是：

- 单 `Activity`
- 单一 `LauncherState`
- 自定义像素渲染链路
- 页面状态机驱动的输入与绘制

运行时主链路是：

1. `MainActivity` 初始化仓库、字体、渲染器、显示视图和输入代理
2. 仓库层从系统服务或持久化中读取真实数据
3. `LauncherStateTransitions` 把输入、系统事件和数据快照收敛成新的 `LauncherState`
4. `PixelRenderer` 根据 `LauncherState + ScreenProfile + AnimationState` 输出 `PixelBuffer`
5. `PixelDisplayView` 把像素缓冲绘制到真实屏幕

这意味着当前应用不是“控件树驱动 UI”，而是“状态驱动像素绘制”。

## 2. 代码分层

当前主代码位于 `app/src/main/kotlin/com/purride/pixellauncherv2`，按职责分成以下几个包：

- `app`
  - 应用入口和主编排层
  - 代表文件：`MainActivity.kt`、`PixelLauncherApp.kt`
- `launcher`
  - 启动器状态、页面布局、抽屉/设置列表模型、搜索与交互辅助
  - 代表文件：`LauncherState.kt`、`LauncherStateTransitions.kt`、`AppListLayout.kt`
- `render`
  - 像素字体、像素缓冲、页面绘制、主题、分辨率、动画
  - 代表文件：`PixelRenderer.kt`、`PixelDisplayView.kt`、`PixelFontEngine.kt`
- `data`
  - 与系统服务、权限、持久化、网络数据源的接线层
  - 代表文件：`PackageManagerAppRepository.kt`、`ScreenUsageRepository.kt`
- `system`
  - 启动 app、窗口模式、重力映射等系统行为封装
- `util`
  - 时间文本、标签格式化、点击节流等轻量工具

推荐的扩展方向仍然是：

- 新数据能力放进 `data`
- 新页面结构和状态映射放进 `launcher`
- 新绘制和主题逻辑放进 `render`
- 尽量不要把新增业务继续堆进 `MainActivity`

## 3. 启动与主循环

### 3.1 MainActivity 的职责

`MainActivity` 是整个应用的运行时编排中心，当前主要承担：

- 初始化仓库：
  - 应用列表
  - 外观设置
  - 启动统计
  - 电池/充电状态
  - 下一次闹钟
  - 屏幕使用时间
  - 通话与短信状态
  - 定位
  - 降雨预测
  - 运动输入
- 初始化渲染链路：
  - `PixelFontResolver`
  - `PixelFontEngine`
  - `PixelRenderer`
  - `PixelDisplayView`
- 恢复设置并同步到状态：
  - 字体
  - 像素形状
  - 点阵大小
  - 主题
  - 抽屉对齐和行为偏好
- 驱动多个 ticker：
  - 时钟刷新
  - 装饰动画刷新
  - 交互动画刷新
  - Idle 进入检查
  - Idle 物理帧推进
- 处理输入：
  - 按键
  - 点击
  - 拖动
  - 左右页面切换
  - 抽屉搜索输入代理

### 3.2 为什么 MainActivity 仍然很重

当前项目还处于“底盘已建立、产品层持续重构”的阶段，所以 `MainActivity` 仍然承担大量编排职责。  
这不代表后续应该继续把逻辑堆进去。未来扩展时，更推荐把它拆成：

- 状态更新：`LauncherStateTransitions`
- 数据读取：`data`
- 布局与 hit-test：`launcher/*Layout`
- 绘制：`PixelRenderer`

## 4. 状态机

### 4.1 单一状态来源

`LauncherState` 是当前 UI 的单一事实来源。  
页面显示什么、抽屉处于什么模式、当前选择了哪项、Home 要展示哪些文本，最终都落在 `LauncherState` 上。

当前核心模式包括：

- `HOME`
- `APP_DRAWER`
- `SETTINGS`
- `DIAGNOSTICS`
- `IDLE`

### 4.2 状态切换层

`LauncherStateTransitions` 负责纯状态级的转换和派生逻辑，例如：

- 页面切换
- 抽屉窗口同步
- 搜索结果更新
- 设置窗口重排
- 电池、闹钟、通信、降雨、屏幕使用时间等文本写回

这层的重要约束是：

- 可以依赖已有 `LauncherState`
- 不直接依赖 Android 视图或系统控件
- 不直接做渲染
- 不直接发系统权限请求

它更接近“可测试的 UI 状态纯逻辑层”。

## 5. 页面实现

### 5.1 Home

Home 目前由 `HomeLayout` 负责布局指标，`PixelRenderer.drawHome` 负责绘制。  
当前主要显示：

- 日期
- 闹钟
- 动态信息行：`CALL / SMS / RAIN`
- 使用时间与打开次数
- 终端状态文案
- 上下文卡片（如格言）

Home 的真实数据来自多个仓库聚合：

- `NextAlarmRepository`
- `CommunicationStatusRepository`
- `RainForecastRepository`
- `ScreenUsageRepository`

### 5.2 App Drawer

抽屉当前已经从“居中选中列表”收敛为“顶部对齐的文本列表”，并支持：

- 状态栏搜索入口
- 搜索输入代理
- 英文/拼音检索支持
- 右侧隐藏快速定位区域
- 惯性滚动与方向优先吸附
- 点击应用名直接启动

相关代码分布：

- 布局：`AppListLayout.kt`
- 搜索与排序：`DrawerSearchSupport.kt`
- 列表基座：`TextListSupport.kt`
- 滚动物理：`DrawerVerticalScrollController.kt`
- 命中与点击决策：`DrawerContentTapResolver.kt`
- 绘制：`PixelRenderer.drawAppDrawer`

### 5.3 Settings

设置页当前已经接到和抽屉一致的文本列表基座上，但交互语义不同：

- 没有可见高亮
- 点击直接生效
- 左右键直接切换设置值
- 超出视口后可纵向滚动浏览

设置项内容由 `SettingsMenuModel` 提供，布局由 `SettingsMenuLayout` 计算，绘制由 `PixelRenderer.drawSettings` 完成。

### 5.4 Diagnostics

Diagnostics 目前保留为高级页面，不在主路径中承担核心产品职责。  
它仍由 `DiagnosticsModel` 提供内容，但后续应继续保持轻量，不要重新膨胀成主页面。

### 5.5 Idle

Idle 仍然是待机态页面，不是系统锁屏替代。  
当前主要能力包括：

- 默认时钟显示
- 充电态显示
- `IdleFluidEngine` 驱动的背景动效
- 运动输入参与待机物理

后续如果要继续扩展媒体态、通知态，也应在当前渲染和状态体系上演进，而不是新增第二套待机系统。

## 6. 文本列表基座

当前抽屉和设置页共享一套文本列表基座，这是最近一轮较重要的技术收敛。

核心目标：

- 统一正文列表视口
- 统一滚动与惯性物理
- 统一半行裁剪可见性
- 统一 hit-test 逻辑
- 统一显式操作前的滚动收口

### 6.1 关键类型

- `TextListViewport`
  - 定义列表正文的可视窗口
- `TextListLayoutMetrics`
  - 包装 viewport，供布局层输出
- `TextListRuntimeState`
  - 表达当前列表内部的滚动状态
- `TextListSupport`
  - 负责 viewport 创建、行位置换算、自然裁剪命中、显式操作前收口

### 6.2 当前语义

- 列表正文从 `listStartIndex - 1` 开始尝试渲染
- 顶部与底部半行由视口裁剪自然出现
- 点击命中按真实滚动偏移换算，不再只命中完整行
- 显式操作前应先清掉残余位移、速度和 settle 目标

抽屉和设置页现在都共享这套正文列表语义，差异只保留在页面侧行为：

- 抽屉点击是启动 App
- 设置点击是立刻应用设置项

## 7. 渲染系统

### 7.1 渲染主链路

渲染主入口是 `PixelRenderer.render`。  
它接收：

- `LauncherState`
- `ScreenProfile`
- `LauncherAnimationState`
- 可选的 `HorizontalPageSnapshot`
- 抽屉/设置页滚动偏移

输出是一个 `PixelBuffer`。

### 7.2 ScreenProfile

`ScreenProfileFactory` 根据真实屏幕尺寸、点阵大小和像素形状生成逻辑屏幕参数。  
它决定：

- 逻辑宽高
- 点大小
- 像素间距
- 页面能够容纳多少文本和图形

这也是当前“分辨率档位”设置真实生效的位置。

### 7.3 PixelDisplayView

`PixelDisplayView` 不负责布局业务，只负责：

- 接收新的 `PixelBuffer`
- 用当前 palette 和 pixel shape 绘制
- 将逻辑触控转换成逻辑坐标
- 把点击、拖动、滑动事件回调给 `MainActivity`

### 7.4 动画

当前动画主要分两类：

- 装饰动画
  - 头部充电 tick
  - Drawer reveal
  - Launch shutter
- 交互动画
  - 抽屉/设置页滚动惯性
  - 横向页面拖动与 settle
  - Idle 物理刷新

`LauncherAnimationState` 负责装饰动画时序，交互动画主要由 `MainActivity` 与相关控制器推进。

## 8. 数据层

### 8.1 应用与启动

- `PackageManagerAppRepository`
  - 读取系统可启动 Activity
  - 过滤掉 launcher 自身
  - 生成 `AppEntry`
  - 优先为中文设备准备更合适的显示 label
- `AndroidAppLauncher`
  - 真正发起应用启动
- `LauncherStatsRepository`
  - 记录启动次数与最近应用

### 8.2 设备与传感器

- `DeviceStatusRepository`
  - 电池电量
  - 充电状态
- `DeviceMotionRepository`
  - 重力与加速度输入
  - 供 Idle 流体使用

### 8.3 Home 相关真实数据

- `NextAlarmRepository`
  - 读取系统下一次闹钟并监听变化
- `CommunicationStatusRepository`
  - 读取未接来电数和未读短信数
  - 通过 `ContentObserver` 监听变化
- `ScreenUsageRepository`
  - 读取当天 Usage Events
  - 生成屏幕使用时长与打开次数
- `DeviceLocationRepository`
  - 读取最近位置或请求一次当前位置
- `RainForecastRepository`
  - 调用 Open-Meteo
  - 取未来 6 小时最早的有效降雨时间

### 8.4 设置与持久化

`FontSettingsRepository` 不只负责字体，也负责整个 UI 外观与行为偏好的持久化，包括：

- 字体
- 像素形状
- 分辨率档位
- 主题
- 抽屉对齐方式
- 是否启用 Idle 页面
- 进入抽屉是否自动进入搜索

## 9. 系统能力与权限

当前真实依赖的权限或特殊能力包括：

- `READ_CALL_LOG`
  - 用于未接来电计数
- `READ_SMS`
  - 用于未读短信计数
- `ACCESS_COARSE_LOCATION` / `ACCESS_FINE_LOCATION`
  - 用于降雨提醒
- Usage Access
  - 用于读取屏幕使用时长和打开次数

这里有两个重要约束：

- Usage Access 不是普通运行时权限，需要跳到系统设置页开启
- 降雨提醒依赖定位；当没有权限或拿不到位置时，Home 会显示 `RAIN LOC`

## 10. 关键数据流

### 10.1 启动

1. `MainActivity.onCreate`
2. 初始化仓库与渲染链路
3. 恢复外观和行为设置
4. 加载应用列表
5. 启动时钟 ticker 与状态监听
6. 首次渲染 Home

### 10.2 Home 刷新

1. ticker 或仓库回调触发
2. `LauncherStateTransitions` 写回闹钟、通信、雨、使用时间等字段
3. `renderCurrentFrame`
4. `PixelRenderer.drawHome`

### 10.3 Drawer 搜索

1. 点击状态栏或输入字符进入搜索
2. `LauncherStateTransitions.updateDrawerQuery`
3. 基于排序、recent boost 和搜索评分生成 `drawerVisibleApps`
4. `PixelRenderer.drawAppDrawer`
5. 点击结果直接启动 app

### 10.4 Settings 调整

1. 点击或左右键命中设置项
2. `MainActivity.changeSettingValue`
3. 写入 `FontSettingsRepository`
4. `LauncherStateTransitions.updateAppearance / updateUiBehavior`
5. 触发 `ScreenProfile` 或主题重建并重绘

### 10.5 Rain / Call / SMS / Usage 更新

1. `onResume` 或对应监听触发刷新
2. 仓库读取真实系统或网络数据
3. 状态转换层写回 `LauncherState`
4. Home 固定信息区重绘

## 11. 已知技术债与后续建议

### 11.1 MainActivity 仍偏重

虽然状态和列表逻辑已经有一定收敛，但 `MainActivity` 仍然承担较多交互编排。  
后续若继续扩展：

- 优先继续把页面逻辑从 `MainActivity` 往 `launcher` 和 `data` 下沉

### 11.2 文档与源码路径

当前主代码目录已经统一为 `src/main/kotlin`。  
后续新增文档和代码引用都应继续使用这一路径口径。

### 11.3 列表基座仍可继续收敛

抽屉与设置页已经共享正文列表基座，但未来如果 Diagnostics 也需要滚动浏览，可以评估是否接入同一套基座，而不是再写第三套列表逻辑。

### 11.4 不建议做的事情

- 不要把主界面改写成 Compose
- 不要让渲染器直接读取系统服务
- 不要把 Android 原生控件树直接混进像素主界面
- 不要在没有状态层设计的前提下直接改 `PixelRenderer`

## 12. 阅读顺序建议

如果是第一次接手这个项目，建议按下面顺序阅读：

1. `MainActivity.kt`
2. `LauncherState.kt`
3. `LauncherStateTransitions.kt`

## 13. 工程接手与安全修改指南

这一节原本独立存在于《工程实现说明（第一版）》，现在已经并入本文档。  
它回答的不是“项目如何运行”，而是“接手之后应该怎么改，才不容易把底盘改坏”。

### 13.1 信息来源层级

接手这个项目时，建议按下面的层级判断“哪份信息是准的”：

1. 当前代码
2. 根目录 `README`
3. 本文档
4. 设计与任务文档
5. `.qoder/repowiki` 的自动生成内容

含义：

- 确认“真实实现”时，优先看 1 到 3
- 确认“设计目标”时，再看 4
- 自动生成内容只当辅助地图，不当事实来源

### 13.2 开发前应该先看什么

如果是第一次进入项目：

1. 读根目录 `README`
2. 读本文档
3. 看 `MainActivity.kt`
4. 看 `LauncherState.kt` 和 `LauncherStateTransitions.kt`
5. 再进入具体模块设计文档

如果是准备改某个功能：

- 先确认它属于哪个 `LauncherMode`
- 再确认它的状态是否应该进入 `LauncherState`
- 再确认它的真实数据是否应该落到某个 `Repository`

如果这三件事还没想清楚，不建议直接改渲染器。

## 14. 推荐的改动落点

### 14.1 改页面结构或页面语义

优先落点：

- `LauncherState.kt`
- `LauncherStateTransitions.kt`
- `launcher/*Layout.kt`

适用场景：

- 页面新增固定信息
- 页面需要新的选择状态
- 抽屉或设置页的窗口语义变化
- 页面显示依赖新的派生字段

### 14.2 改真实数据来源

优先落点：

- `data/*Repository.kt`

适用场景：

- 接新权限
- 接系统服务
- 接天气 / 媒体 / 通知 / 通话 / 短信 / Usage Access
- 增加持久化设置项

原则：

- 不要让 `PixelRenderer` 直接读取系统服务
- 不要把系统查询逻辑直接写进 `MainActivity`

### 14.3 改渲染或视觉表现

优先落点：

- `PixelRenderer.kt`
- `render/*`

适用场景：

- 页面排版变化
- 字体和绘制效果变化
- 动画表现变化
- 主题与像素表现变化

原则：

- 渲染层只消费状态和布局指标
- 不要在渲染层临时计算业务状态

### 14.4 改列表滚动、命中和惯性

优先落点：

- `TextListSupport.kt`
- `DrawerVerticalScrollController.kt`
- `AppListLayout.kt`
- `SettingsMenuLayout.kt`

原则：

- 抽屉和设置页已经共享文本列表基座
- 改其中一页的列表行为时，先判断是否应该提升到共享基座
- 不要重新造第三套列表滚动逻辑

## 15. 典型改动流程

推荐按下面顺序开发：

1. 先明确这是“实现真相”变更还是“产品目标”变更
2. 若是产品目标变更，先同步设计文档
3. 定义状态字段和数据来源
4. 接布局指标与 hit-test
5. 最后接 `PixelRenderer`
6. 补最小必要测试
7. 回头同步 `README`、本文档和完成度文档

这个顺序的核心是：  
先决定状态和数据，再决定怎么画，而不是先把 UI 画出来再补状态。

## 16. 测试与验证建议

### 16.1 最低验证

每次改动至少应做：

- `bash ./gradlew :app:compileDebugKotlin`

### 16.2 涉及状态、布局、搜索时

建议补或执行：

- 状态切换测试
- 布局指标测试
- 搜索匹配和排序测试

## 17. 常见易错点

### 17.1 误把设计文档当当前实现

很多设计文档记录的是目标，不一定等于已落地实现。  
开工前如果要确认现状，请先回到根目录 `README` 和本文档。

### 17.2 只改渲染，不改状态

这是当前项目最容易出现的错误。  
如果页面新增了信息，但 `LauncherState` 没有对应来源，后面一定会出现刷新和回退混乱。

### 17.3 只改 MainActivity，不收敛到模块层

短期能跑，长期会拖垮可维护性。  
尤其是列表、搜索、设置、权限和真实数据接入，应该持续往 `launcher` 与 `data` 下沉。

### 17.4 把抽屉和设置页当两套列表系统

这两页现在已经共享文本列表基座。

## 18. 文档同步规则

当发生下面这些变更时，必须同步文档：

- 代码主入口变化：同步 `README` 和本文档
- 架构分层变化：同步本文档
- 产品目标变化：同步《产品总规约》与模块设计文档
- 模块已落地行为变化：同步“完成度与待办清单”类文档
