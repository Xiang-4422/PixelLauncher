# 应用抽屉设计需求（第一版）

本文档用于定义 `Drawer` 模块的设计目标，并同步当前真实实现状态。  
阅读时请区分：

- “设计目标”是产品希望抽屉最终达到的体验
- “当前已实现”是代码里已经成立的真实行为

当前真实实现请同时参考《应用抽屉完成度与待办清单》、根目录 `README` 和《技术实现总览（第一版）》。

## 1. 模块定位

- Drawer 不是浏览页，而是检索工具
- 唯一目标是让用户用尽量少的操作定位并启动目标 App
- 风格保持纯文本、低噪音、像素化
- 不回退到传统图标宫格

## 2. 设计目标

### 2.1 核心原则

- 搜索优先
- 名称优先
- 纯文本优先
- 低操作成本优先
- 已知目标 App 的路径必须短于无目标浏览

### 2.2 目标结构

- 抽屉应以文本结果列表为主
- 搜索应成为主路径
- 未输入时可进行快速定位
- 不显示图标
- 不做最近使用区、收藏区、分类区
- 不做内容消费和推荐

### 2.3 搜索目标

- 支持中文名称
- 支持拼音全拼
- 支持拼音首字母
- 支持英文名称
- 支持英文别名

匹配优先级目标：

1. 前缀精确匹配
2. 名称前缀匹配
3. 拼音前缀匹配
4. 首字母匹配
5. 其他弱匹配

### 2.4 排序目标

- 默认基础序是字母序
- 使用频率修正只做轻度前移
- 搜索结果先看匹配质量，再看 recent 修正
- 不引入最近打开专门分区

## 3. 当前已实现

截至当前代码，Drawer 已经具备这些真实能力：

- 抽屉已经改成顶部对齐的纯文本列表
- 列表正文已经接入共享文本列表基座
- 状态栏承担搜索入口，不再使用独立搜索框行
- 点击状态栏即可进入搜索态
- 搜索态由整条状态栏承担 query 显示
- 抽屉可以直接调起系统输入法做代理输入
- 输入已限制为 ASCII 字符
- 已支持：
  - 中文名称匹配
  - 拼音全拼匹配
  - 拼音首字母匹配
  - 英文名称和英文别名匹配
- 已实现“字母序 + 轻量 recent 修正”
- 已支持隐藏快速定位区
- 可见右侧字母 rail UI 已移除
- 列表支持：
  - 惯性滚动
  - 方向优先吸附
  - 显式操作前的滚动收口
  - 自然裁剪的半行显示
- 点击应用名会直接启动
- 应用名不再保留可见选中高亮
- 抽屉支持左、中、右三种文本对齐方式
- 已支持“进入抽屉后自动进入搜索态”的设置项

## 4. 当前未实现与差距

当前 Drawer 距离设计目标仍有这些差距：

- 仍未落地正式像素输入层，当前还是系统输入法代理
- 搜索命中高亮还没有设计完成
- 抽屉的文档和设置项仍有部分历史表述需要持续清理
- 仍没有更完整的空态和失败态视觉统一方案

## 5. 当前实现口径

当前如果把 Drawer 作为示例工程中的一个模块介绍，推荐这样表述：

- Drawer 已经从“浏览页”收敛成“搜索优先的文本检索页”
- 当前搜索入口已经集成进状态栏
- 当前的快速定位能力来自“右侧隐藏滑动区”，而不是可见字母 rail
- 列表滚动、吸附、命中和设置页已经共享同一套文本列表基座

## 6. 后续演进建议

建议优先按下面顺序继续推进：

1. 完成正式像素输入层
2. 打磨搜索命中高亮与空态
3. 继续收紧搜索排序与结果表现
4. 最后再扩展更细的抽屉自定义能力

## 7. 验收口径

当前 Drawer 如果要被判定为“模块成立”，至少应满足：

- 用户进入抽屉后可以立刻开始检索
- 已知目标 App 时，主要路径依赖搜索而不是长距离滚动
- 纯文本界面仍然足够快，不因去图标而明显降低命中效率
- 列表滚动和点击启动在交互上是稳定的，不需要二次确认
