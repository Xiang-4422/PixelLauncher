# 工程分工与工作包 v1

本文档用于把现有 PRD 转成可直接分配给工程师的工作包。

若团队规模变化，可以调整人员映射，但不建议打乱工作包的依赖顺序。

## 1. 使用方式

- 项目经理或主程先按本文档建立任务卡
- 工程师领任务时，必须同时阅读本文档和对应模块 PRD
- 每个工作包都必须有明确的完成定义
- 未满足完成定义，不允许标记为完成

## 2. 建议团队编制

若当前是最小可用团队，建议按 3 人并行推进：

- 工程师 A：状态壳层、Home、Settings
- 工程师 B：Drawer 搜索、输入、排序
- 工程师 C：Idle、通知、数据接入

若只有 2 人：

- 工程师 A：Home、Settings、全局状态与壳层
- 工程师 B：Drawer、Idle、通知与数据接入

若只有 1 人：

- 严格按本文档的优先级串行推进，不要并开多个大包

## 3. 总体依赖

实施顺序建议：

1. W0 基线清理与壳层对齐
2. W1 Home 壳层与固定信息区
3. W2 Drawer 检索模型
4. W3 Idle 状态壳层
5. W4 数据接入基础层
6. W5 Home 上下文堆叠与通知横幅
7. W6 Drawer 最终搜索体验
8. W7 Idle 状态内容完善
9. W8 Settings 单页系统
10. W9 异常态、联调与收口

说明：

- W0 不完成，后续包不要正式开工
- W4 是 Home 和 Idle 的关键依赖
- W8 必须在主要模块稳定后再接入，不要反过来先做设置页

## 4. 工作包定义

### W0 基线清理与壳层对齐

目标：

- 从产品主路径移除 Boot 自检角色
- 明确 Diagnostics 只保留在 Advanced
- 对齐当前状态机和新产品页面职责

建议负责人：

- 工程师 A

依赖：

- 无

输入文档：

- `product-spec-v1.md`
- `mvp-task-breakdown.md`

输出：

- 主路径状态图
- 启动流程说明
- Diagnostics 降级方案
- Boot 退场方案

完成定义：

- 主流程中不再出现 fake diagnostics 产品体验
- Home、Drawer、Idle、Settings 的状态职责被明确映射到现有状态机
- 团队对“哪些旧行为必须删、哪些暂时保留”达成一致

禁止项：

- 不要在此阶段重写渲染器
- 不要引入新的页面状态膨胀

### W1 Home 壳层与固定信息区

目标：

- 落地 Home 的双组件结构
- 先完成顶部固定信息区的布局框架

建议负责人：

- 工程师 A

依赖：

- W0

输入文档：

- `home-prd-v1.md`
- `product-spec-v1.md`

输出：

- Home 双组件布局骨架
- 固定信息区的布局网格
- 时间、日期、闹钟、电话短信计数、雨天提醒、24 小时轨迹、昨日亮屏次数的占位态

完成定义：

- Home 主体严格只有两个组件
- 固定信息区视觉层级明确
- 即便全部数据缺失，页面仍有稳定的占位表现

禁止项：

- 不要在这一步加入堆叠区复杂交互
- 不要加入快捷入口或最近使用 App

### W2 Drawer 检索模型

目标：

- 把现有分页浏览 Drawer 改造成搜索优先的检索页

建议负责人：

- 工程师 B

依赖：

- W0

输入文档：

- `drawer-prd-v1.md`
- `product-spec-v1.md`

输出：

- Drawer 新布局
- 顶部搜索框
- 默认列表
- 未输入时字母索引
- 基础匹配和排序逻辑

完成定义：

- 进入 Drawer 后可直接开始检索
- 未输入时可用字母索引跳转
- 输入后索引隐藏，只保留结果列表
- 结果排序满足“前缀优先 + 频率轻修正”

禁止项：

- 不要回退到图标宫格
- 不要提前加入分类、固定置顶、语音搜索

### W3 Idle 状态壳层

目标：

- 把现有 Idle 从粒子屏保扩展为状态驱动待机页壳层

建议负责人：

- 工程师 C

依赖：

- W0

输入文档：

- `idle-prd-v1.md`
- `product-spec-v1.md`

输出：

- Idle 状态枚举和切换框架
- 默认态壳层
- 充电态壳层
- 媒体态壳层
- 通知态壳层

完成定义：

- Idle 能根据状态自动切换
- 默认 / 充电 / 媒体 / 通知 四态均有独立展示入口
- 切换结束后能正确回落到当前最高有效状态

禁止项：

- 不要在这一步实现复杂环境感知
- 不要在这一步加入锁屏替代逻辑

### W4 数据接入基础层

目标：

- 打通 MVP 首批必须接入的数据源

建议负责人：

- 工程师 C

依赖：

- W0

输入文档：

- `product-spec-v1.md`
- `mvp-task-breakdown.md`

输出：

- 闹钟数据接入
- 电话数据接入
- 短信数据接入
- 通知读取
- 媒体状态读取
- 电池数据映射
- 屏幕使用轨迹与昨日亮屏次数数据定义
- 天气降雨提醒接入方案
- 格言源接入方案

完成定义：

- 每个数据源都明确权限、刷新频率、空态、失败回退
- Home 和 Idle 所需数据都有统一来源
- 数据接入不直接把系统原始格式暴露给渲染层

禁止项：

- 不要把数据抓取逻辑散落到 UI 层
- 不要在这一步引入 TODO 正式系统

### W5 Home 上下文堆叠与通知横幅

目标：

- 完成 Home 的第二组件
- 完成 Home 的通知表现模型

建议负责人：

- 工程师 A

依赖：

- W1
- W4

输入文档：

- `home-prd-v1.md`
- `product-spec-v1.md`

输出：

- 堆叠区卡片容器

- 格言卡
- 媒体卡
- 通知摘要卡
- TODO 占位卡
- Home 通知横幅与极简标记

完成定义：

禁止项：

- 不要把横幅做成完整通知中心
- 不要让堆叠区变成多卡并列仪表盘

### W6 Drawer 最终搜索体验

目标：

- 完成 Drawer 的搜索体验收口

建议负责人：

- 工程师 B

依赖：

- W2
- W4

输入文档：

- `drawer-prd-v1.md`
- `product-spec-v1.md`

输出：

- 中文、拼音、首字母、英文别名匹配
- 搜索结果空态
- 选中态和命中态表现
- 系统键盘过渡策略

- 空态、弱匹配、多音字误差均有可接受表现

禁止项：

- 不要为了快而牺牲规则一致性
- 不要把频率修正做成明显破坏字母认知的重排序

## 7. 交付要求

每个工作包交付时，必须附带：

- 对应 PRD 引用
- 完成范围说明
- 未完成项列表
- 已知风险
- 验收截图或录屏

## 8. 合并要求

- 先合并壳层，再合并数据接入，再合并表现层收口
- 每个工作包尽量独立提交，不要把多个大包混进一个提交
- 若发现产品文档与实现冲突，应先回到设计文档修订，不要直接在代码里偷偷偏离
