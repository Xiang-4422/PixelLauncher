# 主页设计

本文档用于定义 `Home` 模块的产品目标，并明确当前实现状态与目标之间的差距。  
阅读时请区分：

- “设计目标”代表 Home 最终想成为什么
- “当前已实现”代表代码里已经落地的真实行为

如需确认代码入口，请同时参考根目录 `README` 和 [技术实现总览](../architecture/技术实现总览.md)。

## 1. 模块定位

- Home 是“必须看信息页”，不是启动器大厅
- Home 的目标是减少为了确认状态而打开 App
- Home 应该优先承载高频、低熵、时间敏感、可一眼读完的信息
- Home 不承载最近使用 App、快捷入口、图标墙、完整通知中心、完整天气详情

## 2. 设计目标

### 2.1 页面结构

- 像素状态栏常驻
- 状态栏承担微型时间显示和电量分隔线表达
- Home 主体区域严格只保留两个组件
- 组件 A：顶部固定信息区
- 组件 B：上下文堆叠区

### 2.2 固定信息区目标

- 去掉状态栏后约占主内容高度的二分之一
- 主信息是时间和日期
- 闹钟以“下一个闹钟”的形式出现
- 未接电话和未读短信只显示数量
- 未来可能下雨时给出提醒
- 使用情况要尽量减少认知负担，避免变成图表仪表盘
- 固定信息区不滚动、不切换、不承载复杂交互

### 2.3 上下文堆叠区目标

- 系统根据上下文决定默认显示哪张卡片
- 用户可以手动切换卡片
- 卡片顺序允许后续在设置中调整
- 无上下文内容时，默认显示格言卡
- 第一批卡片类型限定为：
  - 格言
  - 媒体
  - 通知摘要
  - TODO

### 2.4 通知目标

- Home 上的通知以临时横幅形式出现
- 多条通知按串行方式播放
- 横幅消失后，只保留极简标记，不保留完整通知中心
- 默认重点支持电话和短信，其他 App 后续通过白名单接入

## 3. 当前已实现

截至当前代码，Home 已经落地的内容包括：

- 状态栏常驻
- 固定信息区已经存在，不再是早期单行状态文案
- 日期已经显示为单行英文格式：`WEEKDAY MONTH DAY`
- `ALARM` 已接入真实系统下一次闹钟
- `CALL` 已接入真实未接来电数
- `SMS` 已接入真实未读短信数
- `RAIN` 已接入未来六小时降雨提醒
- `USE / OPEN` 已接入真实 Usage Access 数据
- 动态信息行已经采用“有数据才显示”策略：
  - `CALL / SMS / RAIN` 同行组合
  - 无有效信息时整行隐藏
- 上下文区已存在，并且当前至少支持格言卡显示
- Home 的真实数据入口已经接好：
  - `NextAlarmRepository`
  - `CommunicationStatusRepository`
  - `RainForecastRepository`
  - `ScreenUsageRepository`

## 4. 当前未实现与差距

当前 Home 距离设计目标仍有这些差距：

- 固定信息区虽然已经成型，但还不是完全稳定的“双组件视觉系统”
- 上下文堆叠区还没有真正落地为“多卡片可切换容器”
- 媒体卡、通知摘要卡、TODO 卡目前都还没有正式接线
- Home 横幅通知模型还没有形成
- “真正已读”这类通知交互能力还没有落地
- 用户自定义 Home 模块、白名单和卡片顺序的设置还没有落地

## 5. 当前实现口径

当前如果把 Home 作为示例工程的一部分来介绍，应以这套口径为准：

- Home 已经是“真实状态信息页”，不再只是占位页
- 当前最重要的固定信息是：
  - 日期
  - 下一次闹钟
  - `CALL / SMS / RAIN`
  - `USE / OPEN`
  - 终端状态文案
- 上下文区目前还处于早期形态，后续空间主要在这里

## 6. 后续演进建议

建议优先按下面顺序继续推进：

1. 完成上下文堆叠区容器
2. 接入通知横幅与极简标记
3. 接媒体卡与 TODO 卡
4. 最后再做 Home 自定义能力

## 7. 验收口径

当前 Home 如果要被判定为“模块成立”，至少应满足：

- 用户亮屏后能一眼读到时间 / 日期 / 闹钟 / 是否有电话短信 / 今天手机用了多久
- 用户不需要进入电话、短信、闹钟、数字健康类 App 才能完成基础状态确认
- Home 视觉上仍然保持克制，不退化成传统 Android 小组件仪表盘
