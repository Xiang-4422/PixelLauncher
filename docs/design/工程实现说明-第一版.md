# 工程实现说明（第一版）

本文档用于帮助后续工程师快速理解当前项目的真实技术结构，以及在不破坏现有底盘的前提下如何落地新的产品设计。

若与 `.qoder/repowiki` 中的自动生成内容冲突，以当前代码和本文档为准。

## 1. 当前架构结论

- 当前项目不是 Compose 应用
- 当前项目不是多 Activity / 多 Fragment 页面架构
- 当前项目的主结构是“单 Activity + 启动器状态机 + 自定义像素渲染链路”
- 后续产品重构应尽量复用这套底盘，而不是推倒重写

## 2. 运行时主入口

主入口是：

- `app/src/main/java/com/purride/pixellauncherv2/app/MainActivity.kt`

`MainActivity` 当前承担以下职责：

- 初始化应用仓库、设置仓库、统计仓库、设备状态仓库、运动仓库
- 初始化字体解析器、像素渲染器、像素显示视图
- 恢复外观设置
- 维护当前 `LauncherState`
- 驱动时钟刷新、动画刷新、待机超时检查、待机物理帧推进
- 处理返回键、触摸和应用启动

工程含义：

- 当前很多编排逻辑仍在 `MainActivity`
- 后续新增能力应优先往“状态 + 数据 + 渲染”三层拆，不要继续把业务逻辑堆进 `MainActivity`

## 3. 状态机

核心文件：

- `app/src/main/java/com/purride/pixellauncherv2/launcher/LauncherState.kt`
- `app/src/main/java/com/purride/pixellauncherv2/launcher/LauncherStateTransitions.kt`

当前状态模式：

- `HOME`
- `APP_DRAWER`
- `SETTINGS`
- `DIAGNOSTICS`
- `IDLE`

状态机特点：

- `LauncherState` 是当前 UI 的单一事实来源
- `LauncherStateTransitions` 负责状态切换和部分派生字段更新
- 当前状态中已经包含应用列表、抽屉选中项、设置选中项、时间、电量、充电状态、统计信息、终端状态文本、Idle 流体状态

工程含义：

- 新产品设计新增的页面信息，优先考虑是否需要进入 `LauncherState`
- 纯渲染细节不要放进状态机
- 真正的数据源不要直接挂到渲染器上

## 4. 渲染链路

核心文件：

- `app/src/main/java/com/purride/pixellauncherv2/render/PixelDisplayView.kt`
- `app/src/main/java/com/purride/pixellauncherv2/render/PixelRenderer.kt`
- `app/src/main/java/com/purride/pixellauncherv2/render/ScreenProfileFactory.kt`
- `app/src/main/java/com/purride/pixellauncherv2/render/PixelBuffer.kt`

当前链路可以理解为：

1. `MainActivity` 持有 `LauncherState`
2. `PixelRenderer` 根据 `LauncherState + ScreenProfile + AnimationState` 生成 `PixelBuffer`
3. `PixelDisplayView` 将 `PixelBuffer` 绘制到真实屏幕

当前页面绘制入口都在 `PixelRenderer` 中：

- `drawHome`
- `drawAppDrawer`
- `drawSettings`
- `drawDiagnostics`
- `drawIdle`

工程含义：

- 当前 UI 不是 Android 常规控件树
- 新页面能力应继续走“布局指标 + 像素绘制”模式
- 不要把系统原生 Widget 直接塞进当前主界面层

## 5. 页面布局与模型

当前布局相关文件：

- `app/src/main/java/com/purride/pixellauncherv2/launcher/HomeLayout.kt`
- `app/src/main/java/com/purride/pixellauncherv2/launcher/AppListLayout.kt`
- `app/src/main/java/com/purride/pixellauncherv2/launcher/SettingsMenuLayout.kt`
- `app/src/main/java/com/purride/pixellauncherv2/launcher/AppDrawerIndexModel.kt`
- `app/src/main/java/com/purride/pixellauncherv2/launcher/SettingsMenuModel.kt`

当前现状：

- 主页仍然非常轻，基本只有状态文案
- 应用抽屉是分页纯文本浏览器，不是搜索页
- 设置页仍是轻量参数面板
- 待机页主要依赖渲染器中的时钟和流体逻辑

工程含义：

- 新产品能力主要会落在这些布局文件和对应的绘制逻辑上
- 现有布局指标计算方式可以复用
- 现有产品结构不能直接满足新 PRD，需要产品层重构

## 6. 数据层与持久化

当前主要数据文件：

- `app/src/main/java/com/purride/pixellauncherv2/data/PackageManagerAppRepository.kt`
- `app/src/main/java/com/purride/pixellauncherv2/data/DeviceStatusRepository.kt`
- `app/src/main/java/com/purride/pixellauncherv2/data/DeviceMotionRepository.kt`
- `app/src/main/java/com/purride/pixellauncherv2/data/LauncherStatsRepository.kt`
- `app/src/main/java/com/purride/pixellauncherv2/data/FontSettingsRepository.kt`

当前真实能力：

- `PackageManagerAppRepository` 负责读取可启动应用，并按本地语言排序
- `DeviceStatusRepository` 负责电池和充电状态
- `DeviceMotionRepository` 负责待机流体所需的运动输入
- `LauncherStatsRepository` 只记录启动次数、最近应用、上次启动应用
- `FontSettingsRepository` 负责字体、像素形状、点阵大小、主题的持久化

当前缺失能力：

- 通知读取
- 电话和短信聚合
- 天气接入
- 媒体会话接入
- 屏幕使用轨迹
- 昨日亮屏次数
- TODO 数据源

工程含义：

- 新的数据能力建议继续落在 `data/` 层
- 不要让 UI 层自己直接读系统服务
- 后续需要补独立仓库或数据聚合器，而不是把逻辑塞进 `MainActivity`

## 7. 动画与待机

核心文件：

- `app/src/main/java/com/purride/pixellauncherv2/render/LauncherAnimationState.kt`
- `app/src/main/java/com/purride/pixellauncherv2/render/IdleFluidEngine.kt`

当前动画分类：

- Boot 启动动画
- Header 充电跳动
- Drawer Reveal
- Launch Shutter
- Idle Fluid

工程含义：

- Boot 现阶段要弱化或退出主路径
- IdleFluid 可以保留为待机背景能力，但不能继续充当整个待机页本身
- 后续新的充电态、通知态、媒体态应在现有动画体系上扩展，而不是另起一套渲染系统

## 8. 现状与新产品需求的主要差距

主页：

- 当前主页只显示终端状态文本
- 新需求要求双组件、时间日期、闹钟、电话短信计数、雨天提醒、使用轨迹等

应用抽屉：

- 当前应用抽屉是分页浏览
- 新需求要求搜索优先、拼音和首字母匹配、字母索引、低噪音纯文本检索

待机页：

- 当前待机页更像时钟加流体屏保
- 新需求要求状态驱动的默认态、充电态、媒体态、通知态

设置系统：

- 当前设置页主要调整显示参数
- 新需求要求完整单页设置系统

## 9. 推荐扩展方式

推荐遵循下面的扩展顺序：

1. 先定义状态和数据模型
2. 再定义布局指标与绘制入口
3. 最后接入真实系统数据和设置项

推荐的代码落点：

- 页面级状态和派生信息：`LauncherState.kt` 与 `LauncherStateTransitions.kt`
- 页面布局指标：`launcher/` 下的各 Layout 文件
- 像素绘制：`PixelRenderer.kt`
- 数据接入与持久化：`data/`
- 系统调用与启动行为：`system/`

## 10. 当前不建议做的事情

- 不要把整个 UI 改写成 Compose
- 不要绕过状态机直接让渲染器读系统服务
- 不要继续把大型业务逻辑堆进 `MainActivity`
- 不要把 Android 原生控件树直接混入像素主界面
- 不要在没有设计文档更新的前提下擅自扩张页面职责

## 11. 当前建议优先级

从工程角度，最合理的顺序是：

1. 先完成状态壳层清理
2. 再完成主页和应用抽屉的结构重做
3. 然后补数据接入
4. 再完成待机页状态内容
5. 最后做设置系统单页化和异常态收口

原因：

- 主页和应用抽屉定义了主路径
- 数据接入是主页和待机页的共同依赖
- 设置系统过早接线会放大返工成本

## 12. 测试建议

当前项目已经有一定的单元测试基础，目录在：

- `app/src/test`

后续新增能力建议至少覆盖：

- 状态切换逻辑
- 布局指标计算
- 搜索匹配和排序规则
- 数据空态和失败回退
- 设置持久化
