## 1. 渲染系统的设计目标

PixelLauncher 的渲染系统不是“用 Android 控件拼页面”，而是“先生成逻辑像素帧，再决定怎么显示这张帧”。

这样做的原因有三个：

1. 项目要保持统一的像素风格，不能让不同页面各自使用不同的控件绘制习惯。
2. 页面布局、字体、图形、动画都需要受同一套逻辑分辨率约束。
3. 后续如果显示后端需要从 CPU Canvas 切到 GPU 合成，页面层不应该全部重写。

所以当前渲染系统的核心思想是：

- 页面只负责生成逻辑像素内容
- 显示后端只负责把逻辑像素内容映射到真实屏幕
- 页面语义和显示技术解耦

## 2. 当前渲染总链路

当前主链路可以概括成下面这条：

1. `MainActivity` 收集当前运行时状态
2. `LauncherStateTransitions` 把输入和系统数据收敛成 `LauncherState`
3. `PixelRenderer` 根据状态输出逻辑像素内容
4. `PixelFrameView` 选择实际显示后端
5. 真实屏幕显示结果

  - [PixelFrameView.kt](/Users/xiangyu/StudioProjects/PixelLauncher/app/src/main/kotlin/com/purride/pixellauncherv2/render/PixelFrameView.kt)

## 3. 逻辑像素层：PixelBuffer

### 3.1 PixelBuffer 的角色

`PixelBuffer` 是当前页面的逻辑像素帧。  
它不关心真实屏幕有多大，也不关心最终是用 Canvas 还是 GL 显示。

它只表达：

- 逻辑宽度
- 逻辑高度
- 每个逻辑像素的值

当前像素值主要分成三类：

- `OFF`
- `ON`
- `ACCENT`

这意味着页面层并不直接处理真实颜色，而是先输出“语义像素”，真正的颜色由调色板决定。

### 3.2 为什么要先输出逻辑像素帧

这样设计的好处是：

- 字体绘制、图形绘制、列表绘制都只依赖逻辑网格
- 切主题时不用重写页面逻辑
- 切显示后端时，页面层几乎不用改
- 后续可以针对特殊页面做局部合成优化，而不是推倒整套页面绘制

## 4. 几何层：逻辑网格如何映射到真实屏幕

### 4.1 共享几何对象

当前几何映射已经统一到：

- [PixelGridGeometry.kt](/Users/xiangyu/StudioProjects/PixelLauncher/app/src/main/kotlin/com/purride/pixellauncherv2/render/PixelGridGeometry.kt)

它负责统一计算：

- `cellSize`
- `originX`
- `originY`
- `dotInset`
- `dotSize`
- 内容区域宽高

### 4.2 为什么要抽共享几何

这是这轮渲染重构里很重要的一点。  
之前 Canvas 和 GL 各自算一套几何，结果很容易出现：

- 触摸映射和显示不一致
- Canvas 路径看起来正常，GL 路径偏移或翻转
- 像素形状边界不一致

现在的约束是：

- Canvas 后端和 GL 后端必须共用同一套逻辑网格几何
- 触摸命中也必须使用同一套几何
- 不允许两个后端各自维护一套独立公式

这能明显减少“看起来是渲染 bug，其实是几何不一致”的问题。

## 5. 页面绘制层：PixelRenderer

### 5.1 它负责什么

`PixelRenderer` 是页面绘制的统一入口。  
它负责把 `LauncherState` 变成一张或多张逻辑像素帧。

核心入口在：

- [PixelRenderer.kt](/Users/xiangyu/StudioProjects/PixelLauncher/app/src/main/kotlin/com/purride/pixellauncherv2/render/PixelRenderer.kt)

主要方法：

- `render(...)`
  - 页面通用完整帧渲染入口
- `renderIdleStatic(...)`
  - `IDLE` 静态底图渲染入口

### 5.2 为什么新增 renderIdleStatic

这是当前渲染架构里最关键的变化之一。

以前 `IDLE` 页和其他页面一样，走的是：

- 主线程整页重绘
- 每次物理变化都重新生成完整 `PixelBuffer`

这对 `IDLE` 不划算，因为 `IDLE` 中真正高频变化的只有流体遮罩。  
时间文本、静态背景、固定装饰并不会每一帧都变。

所以现在把 `IDLE` 分成两层：

- 静态底图
- 高频动态遮罩

`renderIdleStatic(...)` 的职责就是只生成低频静态内容，不再把流体粒子画进去。

---

## 6. 显示后端抽象：PixelFrameView

### 6.1 为什么要有这一层

`PixelFrameView` 的作用是把“页面如何生成帧”和“帧如何显示”拆开。

它当前暴露的关键能力包括：

- `submitFrame(...)`
- `submitIdleMask(...)`
- `setIdleContinuousRendering(...)`

这几个接口背后的含义是：

- 页面层可以提交完整静态帧
- `IDLE` 动态层可以单独提交遮罩
- 显示层可以在需要时切到连续渲染

### 6.2 这层带来的价值

这层抽象让后续工程师可以在不改页面逻辑的前提下：

- 切换显示后端
- 给某些页面单独加 GPU 合成
- 做特定页面的局部渲染优化

如果没有这层，`IDLE` 的 GL 分层会被迫改写大量 `MainActivity` 和 `PixelRenderer` 主路径。

---

## 7. Canvas 后端：PixelDisplayView

### 7.1 角色

`PixelDisplayView` 是最直接的 CPU 后端。  
它会把 `PixelBuffer` 中的每个逻辑像素逐个画到 `Canvas` 上。

文件：

- [PixelDisplayView.kt](/Users/xiangyu/StudioProjects/PixelLauncher/app/src/main/kotlin/com/purride/pixellauncherv2/render/PixelDisplayView.kt)

### 7.2 特点

优点：

- 逻辑简单
- 容易调试
- 作为 fallback 很稳

缺点：

- 每帧需要逐像素绘制
- 对高频动画不够友好
- 当页面内容和像素数量上来后，CPU 压力明显

### 7.3 当前定位

Canvas 后端现在更像：

- 设备不支持 GLES2 时的 fallback
- 逻辑等价性的参考实现
- 新渲染方案调试时的对照基线

它不再是 `IDLE` 高频动画的最佳路径。

---

## 8. GL 后端：PixelGlDisplayView + PixelGlRenderer

### 8.1 两层角色分工

- `PixelGlDisplayView`
  - 视图容器
  - 负责和 `GLSurfaceView` 生命周期对接
  - 负责接收完整帧和 Idle 遮罩
- `PixelGlRenderer`
  - 真正的 GL 渲染器
  - 负责纹理上传、uniform 更新和最终合成绘制

### 8.2 为什么需要 GL 后端

引入 GL 的原因，不是为了“炫技”，而是为了两个现实问题：

1. Canvas 路径在高频动画下 CPU 成本高
2. `IDLE` 需要更适合动态图层的显示模型

所以 GL 的核心意义是：

- 让最终显示和合成交给 GPU
- 让静态底图和动态遮罩可以分开更新

### 8.3 当前 GL 路径做了什么

GL 路径当前已经具备：

- 完整静态帧纹理上传
- `IDLE` 动态遮罩纹理上传
- 静态底图 + 遮罩合成
- `IDLE` 期间连续渲染
- 非 `IDLE` 页面按需渲染

这意味着：

- 非 `IDLE` 页面仍然按“需要时更新整帧”工作
- `IDLE` 页面可以单独高频刷新遮罩，而不必每一帧都重建整页

## 9. Idle 的专项渲染改造

### 9.1 改造前的问题

改造前 `IDLE` 的主要问题是：

- 物理线程每推进一次
- 主线程就要重建完整 `PixelBuffer`
- 显示层再整页提交

这会把“高频变化的流体层”和“低频变化的静态 UI”绑死在一起。

### 9.2 改造后的结构

现在 `IDLE` 变成：

- `Idle static frame`
  - 低频更新
- `Idle mask frame`
  - 高频更新
- `GL compositor`
  - 实时合成显示

关键类型：

- [IdleMaskFrame.kt](/Users/xiangyu/StudioProjects/PixelLauncher/app/src/main/kotlin/com/purride/pixellauncherv2/render/IdleMaskFrame.kt)
- [FrameSwapBuffer.kt](/Users/xiangyu/StudioProjects/PixelLauncher/app/src/main/kotlin/com/purride/pixellauncherv2/render/FrameSwapBuffer.kt)

### 9.3 当前动态遮罩的语义

`IdleMaskFrame.mask` 当前是单通道数据：

- `0x00` 表示不点亮
- `0x7F` 表示点亮

GL 合成时的规则是：

- 先取静态底图颜色
- 再看对应逻辑像素上的遮罩值
- 如果遮罩点亮，并且静态底图当前位置本身是 `OFF`
- 则用 `pixelOnColor` 覆盖

这样可以保证：

- 流体层能亮起来
- 但不会覆盖时间文本这类已经点亮的静态信息

### 9.4 为什么不直接上传整张 Idle 完整帧

因为那样仍然会回到旧问题：

- 每次物理 step 都要整页重建
- 高频更新和低频静态内容没有解耦

遮罩单独上传的好处是：

- 每帧变动数据更小
- 主线程不需要反复排版静态文本
- `IDLE` 的高频刷新真正只集中在动态层

## 10. Idle 物理线程和主线程的职责分离

### 10.1 当前线程模型

现在 `MainActivity` 中的 `IDLE` 链路已经收成下面这种职责分配：

- 物理线程
  - 运行 `IdleFluidEngine`
  - 生成最新 `IdleMaskFrame`
- 主线程
  - 只在低频变化时重建 `IDLE` 静态底图
- GL 线程
  - 按连续渲染节奏合成显示

### 10.2 为什么这比旧模型更合理

旧模型的问题是：

- 物理线程只负责算
- 主线程还要承担高频重绘

现在的好处是：

- 物理线程负责高频数据生产
- 主线程只做低频 UI 更新
- GL 线程负责最终显示

这更接近现代渲染系统里常见的“静态层 + 动态层 + 合成器”的结构。

## 11. 坐标映射与 GL 等价性

### 11.1 之前出过什么问题

GL 路径初版曾出现过两个典型问题：

1. 逻辑帧的 `Y` 方向和 Canvas 路径不一致
2. 单通道纹理上传时没有设置正确的 unpack alignment，导致画面出现斜切错位

第二个问题尤其隐蔽。  
单字节纹理如果仍然使用默认 `GL_UNPACK_ALIGNMENT = 4`，当纹理宽度不是 `4` 的倍数时，每一行数据都会错位，视觉上就会像整屏倾斜了一样。

### 11.2 当前怎么避免这类问题

当前已经采用两条约束：

- 共享几何必须来自 `PixelGridGeometry`
- 单通道纹理上传前必须设置：
  - `glPixelStorei(GL_UNPACK_ALIGNMENT, 1)`

这两条属于“以后不要再删掉”的基础约束。

## 12. 为什么当前性能瓶颈不在 GL 上传

这轮实际加了 `RenderPerf` 聚合日志后，已经可以更清楚地区分问题来源。

当前日志说明：

- `IDLE` 静态帧渲染很轻
- 遮罩编码和提交很轻
- Idle 遮罩纹理上传也很轻
- GL 合成接近 60fps

真正重的是：

- `IdleFluidEngine.syncToBattery(...)`
- `IdleFluidEngine.step(...)`
- 其中最可疑的是 coverage 构建和粒子步进

也就是说，当前 `IDLE` 动画的体感低帧率，已经不再主要是“显示层太慢”，而是“物理更新太慢”。

这正是为什么渲染分层是必要的：

- 它先把显示瓶颈和物理瓶颈拆开
- 后续工程师就不会继续在错误方向上盲目优化

## 13. RenderPerf 日志是怎么用的

文件：

- [RenderPerfLogger.kt](/Users/xiangyu/StudioProjects/PixelLauncher/app/src/main/kotlin/com/purride/pixellauncherv2/render/RenderPerfLogger.kt)

它的设计原则是：

- 不逐帧刷屏
- 按时间窗口聚合
- 看平均值、最大值和频率

当前重点埋点包括：

- 主线程完整帧渲染
- Idle 静态帧渲染
- GL 完整帧提交
- GL 遮罩提交
- GL 纹理上传
- GL 合成绘制
- Idle 物理 step
- Idle 遮罩编码

这套日志的价值不是“证明某段代码跑过了”，而是帮助我们回答：

- 主线程是不是还在高频重绘
- 动态层提交是不是太重
- GL 合成是不是已经成为瓶颈
- 物理 step 到底慢在哪

## 14. 后续优化应该优先做什么

如果后面继续优化 `IDLE` 动画，建议顺序如下：

### 第一优先：继续细分 Idle 物理阶段

重点继续拆：

- `world.step(...)`
- `buildCoverageField(...)`
- `buildLitMask(...)`

因为现在最大的成本已经集中在 `IdleFluidEngine` 内部。

### 第二优先：优化 coverage 构建

当前 coverage 的实现特征是：

- 遍历粒子
- 遍历局部区域
- 做距离计算

这很可能是最大的 CPU 热点。  
典型优化方向包括：

- 预计算 kernel
- 避免循环中的 `sqrt`
- 减少 splat 区域开销

### 第三优先：保住当前分层边界

后续优化时不要轻易回退这些边界：

- 不要把 `IDLE` 高频动画重新塞回主线程整页重绘
- 不要让页面层重新关心 GL 细节
- 不要让 Canvas 和 GL 再次各算一套几何

---

## 15. 哪些地方不要轻易改

下面这些是当前渲染体系里比较关键的结构点，不建议随意破坏：

- `PixelRenderer` 仍然是页面像素内容的统一入口
- `PixelFrameView` 是显示后端抽象边界
- `PixelGridGeometry` 是 Canvas / GL / 触摸映射共享几何
- `IDLE` 的静态底图和动态遮罩必须继续解耦
- `RenderPerfLogger` 的聚合日志方式比逐帧日志更适合长期定位性能问题

如果要改这些地方，建议先回答两个问题：

1. 这个改动是在减少层次，还是重新把层次缠回去？
2. 这个改动是在解决真实瓶颈，还是只是把成本转移到别的线程？

---

## 16. 一句话总结

当前 PixelLauncher 的渲染系统已经从“单一 CPU 整页绘制”演进到“逻辑像素帧 + 显示后端抽象 + Idle 动态层专项合成”的结构。

这套结构最重要的价值不是“已经完全最优”，而是：

- 它把页面绘制、显示后端和高频动画更新分开了
- 它让我们能用日志清楚地区分显示瓶颈和物理瓶颈
- 它为后续继续优化 `IDLE` 动画留下了正确的技术方向
