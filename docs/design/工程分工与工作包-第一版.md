# 工程分工与工作包（第一版）

本文档用于把现有产品文档转成当前工程师可直接执行的工作包。

当前项目只有一名工程师，因此本文档只保留单工程师串行方案，不再维护多人并行分工。

## 1. 使用方式

- 开工前先阅读《产品总规约（第一版）》和《工程实现说明（第一版）》
- 进入某个工作包前，再阅读对应模块的设计文档
- 每个工作包都必须有明确的完成定义和禁止项
- 未达到完成定义，不允许标记为完成

## 2. 当前执行模式

- 当前只有一名工程师
- 所有工作包按依赖顺序串行推进
- 不建议同时开启两个以上的大工作包
- 若中途插入新需求，先回到产品文档修订，再调整工作顺序

## 3. 总体执行顺序

1. W0 基线清理与壳层对齐
2. W1 主页壳层与固定信息区
3. W2 应用抽屉检索模型
4. W3 待机页状态壳层
5. W4 数据接入基础层
6. W5 主页上下文堆叠与通知横幅
7. W6 应用抽屉最终搜索体验
8. W7 待机页状态内容完善
9. W8 设置系统单页化
10. W9 异常态、联调与收口

执行原则：

- W0 未完成前，不正式进入后续工作包
- W4 是主页和待机页的关键依赖
- W8 不要提前到前面做，必须在主要页面稳定后接入

## 4. 工作包定义

### W0 基线清理与壳层对齐

目标：

- 从产品主路径移除 Boot 自检角色
- 明确 Diagnostics 只保留在 Advanced
- 对齐当前状态机和新的页面职责

负责人：

- 当前唯一工程师

依赖：

- 无

输入文档：

- `产品总规约-第一版.md`
- `最小可用版本任务拆分.md`
- `工程实现说明-第一版.md`

输出：

- 主路径状态图
- 启动流程说明
- Diagnostics 降级方案
- Boot 退场方案

完成定义：

- 主流程中不再出现 fake diagnostics 产品体验
- 主页、应用抽屉、待机页、设置系统的职责能映射到现有状态机
- 团队对“哪些旧行为必须删、哪些暂时保留”有明确结论

禁止项：

- 不要在此阶段重写渲染器
- 不要在此阶段引入新的页面状态膨胀

## 7. 合并要求

- 先合并壳层，再合并数据接入，再合并表现层收口

- 若发现设计文档与实现冲突，应先回到设计文档修订，不要直接在代码里偷偷偏离
